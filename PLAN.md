## 7. UI를 대폭 수정하기
7-1 오른쪽 패널과 하단 버튼을 통합하여 MAIN페이지를 제작한다.
```
<widget class="QFrame" name="rightPanel">
<layout class="QHBoxLayout" name="buttonLayout">
```
이후 일기 목록에서 일기를 선택한다면, 선택한 페이지가 등장하도록 수정한다.

**검토**
- `rightPanel`, `buttonLayout` 위젯명 확인됨(`ui/main_window.ui:206`, `:300`). 지금 `buttonLayout`은 `rightPanel` **안에** 이미 포함된 하단 버튼 줄(저장/삭제/키워드 분석/새 일기)이라, "오른쪽 패널과 하단 버튼을 통합"이 정확히 뭘 바꾸겠다는 건지(이미 한 프레임 안에 있음) — 레이아웃 구조 변경인지, 아니면 이 버튼들을 다른 위치(예: 목록 선택 시 나타나는 페이지 쪽)로 옮기겠다는 뜻인지 확인이 필요함.
- "선택한 페이지가 등장하도록 수정" — 지금은 목록에서 일기를 클릭하면 **같은 입력 폼**(`_on_diary_selected`)에 내용을 채워 넣어 바로 수정 가능한 상태로 보여준다(별도의 "보기 전용 페이지"가 없음). 이 항목이 원하는 게 (a) 지금처럼 같은 폼에 로드하되 레이아웃만 재배치하는 것인지, 아니면 (b) 목록 선택 시 "보기 전용 상세 페이지"로 전환되고 수정하려면 별도 동작이 필요한 **모드 전환 UI**(예: `QStackedWidget`으로 "목록/작성 페이지" ↔ "상세 보기 페이지" 전환)를 새로 만드는 것인지에 따라 작업 난이도가 크게 달라짐. 후자라면 지금까지 없던 새로운 화면 상태(뷰 모드 vs 편집 모드)를 설계해야 함.
- **4번(비밀 일기장 UI, 구현 완료)과 맞물림**: 비밀 일기장 화면이 메인 화면과 완전히 분리된 별도 창/다이얼로그일지, 아니면 여기서 만들 페이지 전환 구조(예: `QStackedWidget`) 안의 한 페이지로 들어갈지는 7번 설계 시 같이 결정하는 게 좋음 — 지금은 "🔒 비밀일기 찾기"/"🚪 나가기" 버튼이 임시 위치(버튼 줄)에 놓여 있는 상태.
- 6번(필터링 적용, 구현 완료)과도 맞물림: 좌측 패널에 "🔍 필터" 접기/펼치기 버튼과 카테고리/학점/위치/키워드 3종/기간 입력창이 이미 추가돼 있음 — 7번에서 레이아웃을 다시 짤 때 이 위젯들도 같이 자리를 잡아야 함.
- 7번은 사실상 이번 목록에서 가장 손이 많이 가는 항목이라, 다른 항목이 만들어낸 위젯(4번의 비밀일기장 버튼, 6번의 필터 UI 등)을 다 반영한 뒤 **마지막에** 레이아웃을 재배치하는 순서가 안전해 보임. 반대로 레이아웃부터 확정하고 그 위에 기능을 얹는 순서를 원한다면, 다른 항목에서 새로 추가되는 위젯들이 어디에 배치될지를 7번 설계 단계에서 미리 자리 잡아둬야 함. (요약 표시는 이미 목록에 반영되어 있음 — 1번 항목 구현 완료.)

**확정된 사항 (2026-07-09 결정)**

- **7-2 MAIN 화면 = 캘린더 — ✅ 결정**: 앱을 처음 실행했을 때 뜨는 MAIN 화면을 8번에서 만들 **캘린더(감정 점수 히트맵) 화면**으로 한다. 이는 검토에서 남겨뒀던 "(a) 같은 폼 재배치 vs (b) 보기 전용 상세 페이지로 전환되는 모드 전환 UI" 질문에 대한 답이기도 함 — **(b) 방향으로 확정**: 캘린더가 첫 화면(엔트리 포인트)이 되고, 날짜(일기 작성일)를 클릭하면 그날의 일기 보기/편집 페이지로 전환되는 화면 전환 구조(예: `QStackedWidget`)가 필요해짐.

- **7-6 비밀 일기장 UI 재설계 여부 — ✅ 결정(작업 불필요)**: 비밀 일기장 UI(4번)는 이미 구성이 끝나 있으므로, 7번에서 이를 좌/우 스택 구조 안에 억지로 통합하는 별도 재설계는 하지 않는다 — 현재의 "🔒 비밀일기 찾기"/"🚪 나가기" 버튼·읽기 전용 모드 방식 그대로 유지.

**확정된 사항 (2026-07-09 결정)**

- **8-1 캘린더 뷰 추가 — ✅ 결정**: 꺾은선 그래프와 별개로, **일기를 작성한 날짜를 달력에서 확인할 수 있는 캘린더(감정 점수 히트맵) 뷰**를 8번 범위에 포함시키기로 함. 날짜 셀마다 일기 작성 여부를 표시하고, `EmotionScore.value`(감정 점수)를 색상으로 함께 보여준다 — 예: 점수가 높을수록(긍정) 따뜻한 색, 낮을수록(부정) 차가운 색으로 칠하는 캘린더 히트맵.

  - **Tk**: 표준 라이브러리에 캘린더 위젯이 없어 직접 그리드(예: `tkinter.Grid`/`Canvas`로 7×N 달력 직접 구현) 또는 `tkcalendar` 같은 외부 패키지 도입 중 선택 필요 — 새 의존성 추가 여부이므로 착수 시 확인.
  - 꺾은선 그래프(시계열 추이)와 캘린더 히트맵(월 단위 개요)은 **서로 다른 관점**이라 상호 대체가 아니라 **함께 제공**하는 방향으로 감(같은 다이얼로그/화면에 탭 또는 영역을 나눠 배치).
  - **7-2와 연동 — ✅ 결정**: 이 캘린더 화면이 단순 보조 시각화가 아니라 **앱의 MAIN 화면(최초 실행 시 뜨는 첫 화면)** 이 되는 것으로 확정됨(`7-2` 참고). 따라서 8번 작업은 "그래프 다이얼로그에 캘린더 탭 추가" 수준이 아니라, **메인 윈도우 자체의 첫 화면을 캘린더로 구성**하는 7번 UI 개편과 통합해서 진행해야 함.

## 9. 해시태그
(원문 그대로 — 상세 요구사항 없음)

**검토**
- 지금 `Diary`/CSV 스키마(`infrastructure/persistence/csv_diary_repository.py`)에는 태그를 담을 컬럼이 없음. 새 컬럼(예: `tags`, 콤마 구분 문자열)을 추가하고 `_row_to_entity`/`_entity_to_row`, 레거시 CSV 정규화 로직(누락 컬럼 기본값 처리)까지 함께 손봐야 함.
- 입력 방식 결정 필요: (a) 사용자가 직접 `#태그` 형식으로 자유 입력, (b) `engine/keyword_analyzer.py`/`TextProcessor`가 이미 추출하는 키워드를 해시태그 후보로 자동 제안, (c) 둘의 혼합(자동 제안 + 수동 추가/삭제). 이미 존재하는 형태소 분석 파이프라인을 재사용하면 (b)/(c) 쪽이 작업량이 적음.
- 저장 후 활용처를 같이 정해야 실익이 있음 — 예: 목록에 태그 칩 표시, `DiaryFilter`(5번, 구현 완료)에 `tag=` 조건 추가, 태그 클릭 시 해당 태그가 달린 일기만 필터링. 즉 9번은 이미 구현된 필터링 구조와 맞물려 설계하는 게 자연스러움.

## 10. 일기장 안에서의 이점 제공
일기를 쓰면 뭔가 이점을 준다.

**검토**
- 원문이 추상적이라 먼저 "이점"의 형태를 정하는 게 우선임 — 예: (a) 연속 작성일수(스트릭) 표시, (b) 작성 횟수에 따른 배지/칭호, (c) 8번 그래프·9번 해시태그와 연동한 통계/리포트 열람 권한, (d) 그림판 브러시나 테마 같은 꾸미기 요소 잠금 해제 등. 방향에 따라 필요한 데이터가 완전히 달라짐.
- 스트릭/횟수 기반이라면 CSV에 이미 있는 `date`/`created_at`만으로 계산 가능해 새 저장소 없이 시작할 수 있음(단, 연속 작성 판정 시 `is_hidden` 일기 포함 여부 결정 필요).
- 배지·잠금해제 같은 영속적 상태가 필요하면 `SecretPasswordStore`/`LocationPresetStore`처럼 작은 전용 저장 파일(예: `data/achievements.json`)이 하나 더 필요함 — "이 프로젝트에 CSV 외의 영속 저장소를 얼마나 늘릴 것인가"라는 공통 질문과 맞닿아 있음.
- 이 항목은 다른 9개 항목과 달리 순수 신규 기능(기존 UI/데이터 흐름 리팩터링이 아님)이라 우선순위·범위를 가장 먼저 명확히 해야 작업 규모를 가늠할 수 있음.
